Google Ads API Access Request

This document explains why PennyLime is applying for a Google Ads API developer token and how the API will be used. We are a first-party advertiser managing our own ad spend; the API will only operate against accounts we directly own and operate.

Use Case Summary

We have built an internal admin dashboard for the PennyLime team to manage advertising and operations in one place. The Google Ads API will let us replace manual exports and stitched-together reporting with live, automated workflows that close the loop between an ad click and a funded loan.

Primary value: Most of our conversions happen offline—a borrower is not approved or funded the same day they click an ad. Without offline conversion uploads, Smart Bidding optimizes against the wrong signal (form submissions instead of approved loans). The API lets us send the true outcome back to Google Ads so bidding optimizes for funded loans.

Specific API Features Used

Endpoint / Service Purpose
conversionUploadService.uploadClickConversions Upload offline conversions when an admin marks a loan APPROVED or FUNDED. We pass the captured gclid, conversion action, value (loan amount), and conversion date-time. This is the core feature we need.
googleAds:searchStream over campaign Daily read-only pull of cost, impressions, clicks, and conversions per campaign for our internal dashboard ROAS / CAC reporting.
conversionActionService (read-only) Resolve conversion-action resource names so the upload step uses the correct action ID per event (lead submit vs. application complete vs. approved vs. funded).

We do not need ad creation, account management, billing, or change-history endpoints. All write operations are limited to offline-conversion uploads on accounts we own.

Architecture & Security

  • OAuth 2.0 with a refresh token issued for the PennyLime Google account; secrets stored encrypted in our infrastructure.
  • Calls originate only from our admin backend; there is no client-side exposure of API credentials.
  • Click-IDs (gclid) are captured first-party at ad click and stored against the contact / loan record so the offline upload can match the exact click that produced the conversion.
  • All uploads are partial-failure aware and idempotent; identical conversions are safely deduplicated by Google Ads via order_id (loan application code).

Compliance

  • We comply with all Google Ads Policies, including the personal-loan policies for the United States.
  • We follow the Required Minimum Functionality guidelines for Basic Access.
  • We do not store PII beyond what is required to operate the lending product, and any data shared with Google (hashed user data for enhanced conversions) is hashed with SHA-256 prior to upload.
  • We do not resell, white-label, or expose this API to third parties.
